The Root Cause

Point-to-point architecture scales quadratically.

Every system stores its own data in its own format. Connecting any two systems requires a custom integration. The number of connections grows with the square of the number of systems.

Each connection is custom-built, manually maintained, and breaks when either system updates. The cost isn't just the initial build — it's the ongoing maintenance, the data inconsistencies, and the questions that never get asked because everyone knows the answer would take too long.

The Insight

What if every machine published its data to one shared layer —
and any system could read it?

Instead of connecting every system to every other system, every system connects to one MQTT broker. Machines publish their state. Applications subscribe to what they need. Adding a new system means one new connection — not N. Adding a new consumer means subscribing to existing topics — no changes to producers required.

That's the Unified Namespace.

The Unified Namespace

One data layer. ISA-95 addressing. MQTT pub/sub.

Every data point gets a canonical address following the ISA-95 international standard. The path is self-describing — any consumer can parse it without mapping tables.

v1.0 / enterprise / site1 / area1 / cnc-01 / status
└─ version
└─ company
└─ factory
└─ area
└─ machine
└─ data point

Producers don't know consumers. Consumers don't know producers. The broker handles delivery. Adding a new machine = publishing to a new topic. The entire pipeline picks it up automatically.

The Architecture

12 functions. 4 layers. One pipeline.

Data flows through infrastructure, not through function calls. Each function runs in its own container, does one job, and communicates through shared MQTT, Valkey, and PostgreSQL. Deploy, update, or replace any function independently.

End-to-End

One event. Seven steps. Seconds.

Follow a single machine event — cnc-01 transitions from ACTIVE to ALARM — through every layer of the pipeline.

1

cnc-01 publishes status: ALARM

uns-sim → MQTT topic v1.0/enterprise/site1/area1/cnc-01/status

2

uns-framework caches the new value

Valkey: uns:data:* = ALARM, uns:prev:* = ACTIVE

3

uns-historian logs the change immediately

PostgreSQL: INSERT INTO uns_historian — real-time, no delay

4

uns-state detects ACTIVE → ALARM transition

Logs: state=ACTIVE, duration_s=120, next_state=ALARM

5

uns-stoppage auto-classifies: ALARM → FAULT

Operator can override to TOOL_BREAK later

6

uns-kpi recalculates: utilisation drops, MTTR starts

Next API call returns updated KPIs reflecting the alarm

7

uns-ai detects anomaly, publishes root cause

"Tool T03 offset trending +0.02mm over 4 hours before alarm"

Layer 4 — AI

The same data, but now it thinks.

uns-ai reads live data from Valkey and historical data from PostgreSQL, assembles rich context, sends it to an AI provider, publishes structured results back to MQTT, and stores a full audit trail. Seven built-in analysis types.

anomaly

Anomaly Detection

Catches patterns humans miss — a machine technically running but cycling abnormally. Compares against 7-day baselines.

shift-summary

Shift Summaries

Replaces manual handover notes. Auto-generated report with utilisation, stoppages, production, and issues per machine.

smart-alert

Smart Alerting

Contextual alert triage — not just "alarm happened" but "this alarm matters because tool life is at 93% and offset is trending".

predictive

Predictive Maintenance

Tool wear prediction from historical patterns. Tells you when to change a tool before it breaks.

optimisation

Production Optimisation

Machine-program assignment based on actual throughput, scrap rates, and reliability data.

root-cause

Root Cause Analysis

Explains why an alarm happened — not just that it happened. Correlates state timeline, tool data, and sensor history.

custom

Custom Prompts

User-defined analysis with configurable data sources. Ask any question about your manufacturing data.

Real Hardware

From PLC to pipeline. Not just software.

The UNS connects to real machines on real factory floors. Wired Ethernet for short runs, wireless LoRaWAN when cabling is impractical.

Wired Ethernet — Direct MQTT

Siemens Logo 8.4 publishes MQTT natively. Ethernet cable → switch → broker. No bridge, no protocol translation. Simplest architecture, lowest latency.

Best for: Short cable runs, new builds, fixed machines, real-time requirements.

Wireless LoRaWAN — When Cabling Isn't Practical

Bridge device at each machine (Advantech WISE-6610 V2) repackages MQTT into LoRa packets. Central gateway (MultiTech Conduit 300) receives and republishes to broker. 2–5km range, licence-free, no subscriptions.

Best for: Retrofit, many machines, monitoring, no existing network drops.

The Ecosystem

Three projects. One unified solution.

An open standard for structuring manufacturing data, an open platform for deploying serverless functions, and a reference implementation with 12 functions across 4 layers.

01 STANDARD

UNS Framework

unsframework.com

An open YAML standard that defines how to structure manufacturing data in MQTT. ISA-95 topic hierarchy, versioned namespaces, sensor templates, producer/consumer definitions. Vendor-neutral — any system that follows the standard can participate.

Defines the data structure
02 PLATFORM

fnkit

fnkit.dev

A lightweight Functions-as-a-Service platform. Write functions in any language — Go, Node.js, Python — scaffold with one command, deploy via git push. Shared cache, database, MQTT broker. No cloud vendor required.

Provides the platform

fn.unsframework.com

Get Started

Stop asking questions you can't answer.

Everything is open source, self-hosted, and ready to deploy. Read the standard, explore the reference implementation, or dive into the documentation.

# Clone and start the UNS pipeline
$ git clone https://github.com/functionkit/fnkit.git && cd fnkit
$ docker network create fnkit-network
$ fnkit cache start
 
# Start capture + process + report
$ cd uns-framework && docker compose up -d && cd ..
$ cd uns-sim && docker compose up -d && cd ..
 
# Verify real-time data
$ curl http://localhost:8080/uns-cache | jq
